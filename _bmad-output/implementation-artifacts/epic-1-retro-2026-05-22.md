## Successes & Strengths

1. **Planning-to-execution pipeline was frictionless** — Detailed Dev Notes with anti-pattern lists, architecture references, and exact implementation patterns eliminated ambiguity. Implementation was nearly mechanical.

2. **Architecture decisions validated early and held throughout** — Tailwind 4 CSS-first config, `@theme inline` vs `@theme`, Astro Fonts API, Content Collections with `glob()` loader — all validated in Stories 1.1–1.2 and carried cleanly through the epic. Zero architectural pivots.

3. **Strict implementation sequencing eliminated rework** — tokens → fonts → content → layout → pages. Each story built cleanly on the previous one. Zero "go back and change Story X" moments.

4. **Accessibility baked into acceptance criteria** — Skip-to-content, `aria-hidden` on decorative elements, 44px tap targets, focus-visible states were in the ACs from the start, not afterthoughts.

5. **Code review process caught meaningful issues** — 12 patches across 7 stories: focus outline visibility, `noreferrer`, tap target width, reduced-motion delay resets, `aria-hidden` on decorative glyphs, sort tiebreakers, `draft` field defaults.

6. **Site works fully without JavaScript** — FR-3 (honest first paint) delivered. Recruiter on mobile sees value proposition, contact link, and project/essay links in first viewport.

7. **Real content shipped, not placeholders** — Sample project and essay content is in LinCie's voice and ready for production.

## Challenges & Growth Areas

1. **20+ deferred items accumulated** — Every code review produced deferred findings. While individually small (schema validation gaps, missing `safe-area-inset`, `og:type` hardcoded), they represent a growing backpack of acknowledged-but-not-fixed work.

2. **Accessibility edge cases slipped to code review** — 3 of 7 stories needed accessibility patches post-implementation: invisible focus outline (1.4), missing `<h1>` for screen readers (1.5), tap target width (1.5). These should have been caught during implementation.

3. **Font budget technically exceeded** — 215KB total vs 200KB target. Accepted because effective first-paint is 57KB (only normal roman loads immediately). Spec-vs-reality gap in how the budget was defined.

## Key Insights & Learnings

1. **The two-pass system (implement → review) works** — but defensive guards (empty strings, missing attributes, contrast logic) could shift left into implementation with a mental checklist.

2. **Anti-pattern documentation prevents syntax bugs** — Tailwind 4 parentheses vs brackets, `ClientRouter` vs `ViewTransitions`, `@theme inline` vs `@theme` — all caught by Dev Notes before becoming issues.

3. **Sequential dependencies + detailed Dev Notes = zero rework** — The combination of strict ordering and comprehensive implementation guidance is the key process insight from Epic 1.

4. **Deferred work is acceptable when documented** — but must be triaged between epics to prevent accumulation becoming a liability.
