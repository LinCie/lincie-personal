## Successes & Strengths

1. **Pattern reuse eliminated rework** — Story 2.1 established the three-column grid, DropCap wrapper, and Colophon placement. Stories 2.2–2.5 reused these patterns verbatim. Zero "go back and change" moments.

2. **Dev Notes quality was exceptional** — Story 2.3 documented the exact remark-generated HTML structure. Story 2.4 included previous-story learnings (tap targets in `rem`, `astro:page-load` pattern). Each story carried forward lessons from the one before.

3. **Zero debug issues across 5 stories** — Every story implemented cleanly on first pass. No TypeScript errors, no build failures, no runtime surprises. Detailed Dev Notes and anti-pattern documentation prevented implementation-time bugs entirely.

4. **Code review hardened components significantly** — Story 2.4 (InlineLink) went from solid to production-grade: `javascript:` URI sanitization, forced-colors support, sr-only new-tab disclosure, malformed URL handling. The review process added real value.

5. **Footnote spike from Epic 1 retro paid off** — The DOM-move approach via `astro:page-load` was clean, correct, and required no rework. Preparation tasks work.

6. **Architecture validated again** — Three-column grid, token system, component boundaries, View Transitions `transition:name` pattern — all held without modification through 5 stories.

7. **Sequential execution continued to prevent rework** — 2.1 (grid + drop cap) → 2.2 (reuse grid) → 2.3 (fill margin column) → 2.4 (link component) → 2.5 (index page). Each story built cleanly on the previous.

## Challenges & Growth Areas

1. **Accessibility edge cases still slipping to review (3 of 5 stories)** — Tap target padding in `em` instead of `rem` (2.3), missing sr-only new-tab disclosure (2.4), `aria-describedby` target hidden by `display: none` (2.3). Subtler than Epic 1's issues but the pattern persists.

2. **Story 2.4 was disproportionately complex** — 8 patches (more than all other stories combined). InlineLink touched security, accessibility, and defensive coding. The story scope may have been too broad for a single unit.

3. **Cross-browser testing is implicit** — Safari `counter(list-item)` reset and forced-colors support were caught in review. No systematic cross-browser checklist exists.

4. **Deferred debt continues to accumulate** — 15 new items this epic. Key items: `opacity: 0` footnotes invisible until Epic 4, desktop→mobile resize breaks footnotes, `font-weight` hover layout shift.

## Key Insights & Learnings

1. **Pattern establishment in early stories pays compound dividends** — Story 2.1's grid pattern was reused 4 times without modification. Investing in getting the first story right is the highest-leverage activity.

2. **Component complexity correlates with patch count** — InlineLink (8 patches) vs ProjectBand (0 patches). Stories with more interaction states need more defensive thinking upfront.

3. **The two-pass system (implement → review) works** — but a11y edge cases need a third mental pass during implementation: "what happens with assistive tech in this interaction state?"

4. **Dev Notes that include previous-story learnings create a knowledge cascade** — Story 2.4 explicitly referenced Story 2.3's tap-target lesson. This prevented the same mistake from recurring.
