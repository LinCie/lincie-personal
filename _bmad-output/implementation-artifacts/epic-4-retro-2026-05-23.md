## Successes & Strengths

1. **First real ScrollTrigger stress test — passed flawlessly** — Epic 3 registered ScrollTrigger but never created an instance. Epic 4 created dozens of instances across two files (`scroll.ts` and `footnotes.ts`), with `normalizeScroll` + pins + fog-lifting + velocity detection all coexisting on the same pages. No conflicts. The architecture held under real load.

2. **`scroll.ts` designed for organic growth** — Story 4.1 created the file with damped scroll only, but structured it to accommodate 4.2 (section pin) and 4.3 (fog-lifting) as extensions. Both stories added cleanly without restructuring. The comment-header section pattern (`// ─── Section Pin ───`) kept the file navigable at ~100 lines.

3. **Correct separation of concerns** — `footnotes.ts` as a separate module from `scroll.ts` was the right architectural call. Independent instance arrays, independent DOM targets, independent lifecycle. No coordination needed between the two files despite both using ScrollTrigger on the same pages.

4. **Zero debug issues — third consecutive epic** — Detailed Dev Notes, anti-pattern documentation, and previous-story learnings sections prevented implementation-time bugs entirely. The knowledge cascade from Epic 2's retro continues to compound.

5. **Code reviews caught real edge cases** — 6 patches applied across Stories 4.2–4.4 addressed genuine issues: pin stuck on last h2, ScrollTrigger.refresh() before fonts load, inline styles not cleaned up, in-flight tweens not tracked, stale HINT_KEY after View Transitions, unbounded array growth. All would have been subtle production bugs.

6. **`document.fonts.ready.then(...)` pattern** — Story 4.2's review discovered that ScrollTrigger.refresh() was calculating pin positions with fallback font metrics. Wrapping in `document.fonts.ready.then(...)` ensures correct positions. This pattern is now available for Epic 5's folio/spine story.

7. **Velocity — entire scroll system in one day** — Four stories delivering FR-23, FR-24, FR-20, and FR-12 in a single session. The accumulated knowledge cascade and proven patterns made this possible.

8. **JS budget well managed** — ~46.5KB gzip total against a 60KB budget. 13.5KB headroom for Epic 5's four scripts.

## Challenges & Growth Areas

1. **Cleanup edge cases are a recurring pattern** — 3 out of 4 stories had review findings about cleanup: inline styles persisting after navigation (4.2), in-flight GSAP tweens not tracked in instance registry (4.3), and `instances[]` accumulating dead references every animation frame (4.4). The gate+cleanup lifecycle handles the happy path but doesn't account for side effects outside the instance registry.

2. **Module-level constants go stale after View Transitions** — `REDUCED_MOTION` (all scripts) and `HINT_KEY` (footnotes.ts) both capture state at module load. After Astro View Transitions change the page without a full reload, these values are stale. `HINT_KEY` was patched; `REDUCED_MOTION` is deferred as an architectural decision.

3. **ScrollTrigger timing sensitivity** — Font loading, DOM moves (FootnoteReveal on `astro:page-load`), and trigger refresh ordering required careful coordination. The `document.fonts.ready.then(...)` wrapping and the explicit ordering documentation in Dev Notes addressed this, but it's inherent complexity in scroll-linked code.

4. **Review patch count increased** — Epic 3 had 0 patches; Epic 4 had 6. This reflects the genuine complexity of scroll-linked interactions (more edge cases than simple timelines), not a quality regression. The reviews are working as intended — catching issues before production.

5. **`--dur-mark` token still not introduced** — Carried from Epic 3 retro. Now three components use hardcoded 250ms (InlineLink, ProjectBand, footnotes.ts). Growing but still low-impact.

## Key Insights & Learnings

1. **Scroll-linked code has fundamentally more edge cases than timeline code** — Timelines are deterministic (fire once, known duration). Scroll-linked code is continuous, bidirectional, and interacts with font loading, DOM mutations, viewport resize, and View Transitions. Code reviews are more valuable on scroll stories.

2. **The instance registry pattern needs a "side effects" companion** — Pushing instances to an array and killing them in cleanup works for GSAP objects. But inline styles, DOM-injected elements, and growing arrays are side effects that need their own cleanup tracking. Stories 4.2–4.4 each solved this ad-hoc (`pinnedHeadings[]`, `foggedElements[]`, `[data-footnote-hint]` selector). A documented pattern would prevent future ad-hoc solutions.

3. **File extension strategy validated** — `scroll.ts` growing from 1 behavior (4.1) to 3 behaviors (4.1 + 4.2 + 4.3) without restructuring proves the section-header pattern works for related behaviors in a single file. `footnotes.ts` as a separate file proves unrelated behaviors should stay separate.

4. **`overwrite: true` is essential for scroll-driven tweens** — Every `gsap.to()` in scroll callbacks uses `overwrite: true` to prevent tween stacking on rapid direction changes. This should be a documented pattern for any future scroll-driven animation.

5. **Previous retro action items compound when followed** — The a11y checklist and cross-browser edge cases sections (from Epic 2 retro) continued to prevent issues. The story status consistency fix (from Epic 3 retro) was followed perfectly. Process improvements stick when they're specific and achievable.
