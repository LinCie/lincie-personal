## Successes & Strengths

1. **Gate+cleanup lifecycle pattern propagated flawlessly** — Story 3.1 established the pattern (imports → gates → instance registry → cleanup → init → lifecycle hooks → initial run). Stories 3.2, 3.3, and 3.4 followed it without deviation. Zero pattern drift across four scripts. Architectural investment in the first story pays compound dividends.

2. **Story sequencing was optimal** — 3.1 (pure infrastructure) → 3.2–3.3 (GSAP scripts proving the pattern) → 3.4 (non-GSAP script still following lifecycle) → 3.5 (pure CSS, no JS). Each story tested a different facet of the architecture without creating blocking dependencies.

3. **Zero debug issues — second consecutive epic** — Detailed Dev Notes, anti-pattern documentation, and previous-story learnings sections prevented implementation-time bugs entirely. The knowledge cascade from Epic 2's retro action items is working.

4. **Zero patches applied in code review** — Epic 2 had 16 patches across 5 stories. Epic 3 had only 4 deferred items (all minor, pre-existing concerns). The upfront documentation is preventing issues rather than catching them in review.

5. **Accessibility gates are comprehensive** — Every GSAP script has dual gates (`REDUCED_MOTION` and `COARSE_POINTER`) evaluated once at module load. Story 3.5 relied on the existing `global.css` reduced-motion safety net for CSS transitions. Clean separation of concerns.

6. **"Tailwind for simple, GSAP for complex" rule validated** — Story 3.5 proved CSS transitions with `var(--ease-mark)` handle hover affordances perfectly. No over-engineering.

7. **Dev Notes quality continued to improve** — Timeline timing breakdown tables (3.2), inertia explanations (3.3), "What Already Exists" tables (3.4), and architecture compliance sections in every story prevented implementation mistakes before they happened.

8. **JS budget well managed** — ~44KB gzip total (GSAP core + ScrollTrigger + 4 scripts) against a 60KB budget. Comfortable headroom for Epic 4's scroll scripts.

## Challenges & Growth Areas

1. **Story file status inconsistency** — Stories 3.2 and 3.3 have `Status: review` in their story files while sprint-status marks them `done`. Minor bookkeeping gap that could confuse future readers.

2. **Missing `--dur-mark` token** — The 250ms hover rhythm is hardcoded in InlineLink.astro and ProjectBand.astro. Falls between existing tokens (`--dur-quick` 150ms, `--dur-breath` 400ms). Two components now have magic numbers instead of a single source of truth.

3. **`globalTimeline.resume()` future interaction** — The unconditional resume on `visibilitychange` in `gsap-init.ts` could conflict with Epic 4's scroll-damping or section-pin pause logic. Not a current defect but a known risk for the next epic.

4. **ScrollTrigger registered but never exercised** — Epic 3 imported and registered ScrollTrigger but never created a single ScrollTrigger instance. The real test comes in Epic 4 where all 4 stories depend on it heavily.

## Key Insights & Learnings

1. **Foundational stories deserve maximum investment** — Story 3.1 was "just" infrastructure (zero animation, zero DOM) but its pattern quality determined the success of every subsequent story. The Epic 2 retro's warning ("if the GSAP init pattern is wrong, every subsequent story inherits the problem") was exactly right.

2. **Not everything needs GSAP** — Story 3.5 (inkstroke underline, hover affordances) was pure CSS transitions. The architecture's complexity threshold is well-calibrated: CSS for state transitions, GSAP for timelines and scroll-linked behavior.

3. **Action items from previous retros compound** — All three Epic 2 action items were delivered and measurably improved Epic 3 quality. The a11y checklist prevented accessibility patches. The cross-browser section caught edge cases upfront. The "split if >5 patches" rule was never triggered because quality was high.

4. **The knowledge cascade accelerates velocity** — Each story's Dev Notes explicitly referenced previous-story learnings. By Story 3.5, the accumulated context meant implementation matched spec on first pass with zero rework.
